# Фронтенд — Доска объявлений Вологодской области

Статический сайт (HTML + CSS + vanilla JS, ES-модули), работающий как SPA
(хэш-роутинг вида `#/ad/42`, что отлично подходит для GitHub Pages — не
нужна серверная настройка rewrite-правил). Общается с бэкендом только через
REST API (`fetch`), без какой-либо сборки (webpack/vite и т.п. не нужны).

## Структура

```
frontend/
  index.html
  css/style.css
  js/
    config.js       # адрес backend API + справочники категорий/городов
    auth.js          # хранение токенов, вход/регистрация/выход, refresh
    api.js            # обёртка над fetch с авто-обновлением access-токена
    router.js          # простой хэш-роутер
    utils.js             # общие хелперы (карточка объявления, форматирование)
    app.js                 # точка входа: роуты, шапка сайта
    views/                  # по одному модулю на экран
```

## Локальный запуск

Файлы используют ES-модули (`<script type="module">`), поэтому открыть
`index.html` напрямую через `file://` не получится — браузер блокирует
модули по CORS-политике для `file://`. Нужен любой локальный HTTP-сервер:

```bash
cd frontend
python3 -m http.server 5500
# затем откройте http://127.0.0.1:5500
```

(Подойдёт и расширение VS Code "Live Server", и `npx serve`.)

Перед этим убедитесь, что бэкенд запущен (см. `backend/README.md`) и что
`js/config.js` указывает на правильный адрес (по умолчанию
`http://127.0.0.1:8000` — подходит для локального запуска бэкенда).

## Настройка перед деплоем

Откройте `js/config.js` и замените:

```js
export const API_BASE_URL = "http://127.0.0.1:8000";
```

на адрес вашего бэкенда на Render, например:

```js
export const API_BASE_URL = "https://vologda-ads-backend.onrender.com";
```

Это единственное обязательное изменение для деплоя.

## Деплой на GitHub Pages

1. Запушьте проект в GitHub-репозиторий (весь проект целиком — и `backend/`,
   и `frontend/` могут жить в одном репозитории).
2. В настройках репозитория: **Settings → Pages**.
3. **Source**: "Deploy from a branch".
4. **Branch**: выберите вашу основную ветку (например `main`) и папку
   **`/frontend`** (GitHub Pages умеет отдавать сайт из подпапки репозитория).
   Если такой опции нет в вашем интерфейсе — самый надёжный вариант:
   создать отдельную ветку `gh-pages`, в которую попадает **только**
   содержимое папки `frontend/` (например, через GitHub Actions или
   `git subtree push`), и указать её как источник Pages.
5. Сохраните — через 1-2 минуты сайт будет доступен по адресу вида
   `https://ваш-логин.github.io/имя-репозитория/`.
6. Вернитесь в настройки бэкенда на Render и укажите точный адрес Pages в
   переменной `CORS_ORIGINS` (например
   `https://ваш-логин.github.io`), иначе браузер будет блокировать запросы
   к API по CORS.

### Простой вариант через GitHub Actions (если папку выбрать нельзя)

Создайте `.github/workflows/deploy-frontend.yml` в корне репозитория:

```yaml
name: Deploy frontend to GitHub Pages
on:
  push:
    branches: [main]
    paths: ["frontend/**"]
jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pages: write
      id-token: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/upload-pages-artifact@v3
        with:
          path: frontend
      - uses: actions/deploy-pages@v4
```

И включите Pages с источником "GitHub Actions" в настройках репозитория.

## Проверка после деплоя

1. Откройте сайт — должна загрузиться главная страница со списком
   объявлений (изначально пустым).
2. Зарегистрируйтесь, создайте объявление, добавьте фото.
3. Откройте `https://ваш-бэкенд.onrender.com/admin/login` — созданное
   объявление должно быть видно в админке (см. `backend/README.md` про
   создание первого администратора).

## Известные ограничения (сознательно, ради простоты проекта)

- Хэш-роутинг (`#/...`) вместо History API — сделано специально, чтобы
  никакой сервер (GitHub Pages) не требовал настройки rewrite-правил для
  SPA — при обычном History API обновление страницы на `/ad/42` без rewrite
  вернёт 404.
- Токены хранятся в `localStorage` (а не в httpOnly-cookie) — это
  стандартный компромисс для SPA, которое обращается к API на другом домене
  без общего родительского домена для cookie.
